What I learned today

The original hand-rolled language parsing / import-normalization direction was getting too deep into compiler and language-server problems. The core insight was that parsing code is only half the problem; semantic resolution is where the real pain starts. Because of that, the pipeline direction shifted toward using SCIP as the semantic substrate instead of continuing to own all low-level cross-language resolution logic manually.

SCIP is language-agnostic as a format, but language-specific in how the data is produced. That means the repo can still support multiple languages, but each language family gets its own indexer wrapper. For now, the important ones are Python and the TypeScript ecosystem. The `scip_indexers/` package exists to wrap those CLIs and normalize how the rest of the pipeline calls them.

The pipeline itself should stay job-oriented: frontend enqueues one repo-analysis job, backend worker consumes it, and one pipeline run handles the full analysis. Internally, the pipeline is made of stages/steps. Those stages should be idempotent, and the pipeline should own retry / failure handling. The queue should stay at the whole-pipeline level for now rather than making every stage its own queued task.

If the project becomes more advanced, the right execution model is a DAG of stage dependencies, not just one flat ordered list forever. The important concept is that stages declare dependency relationships, the scheduler builds the graph from those declarations, validates it, and runs ready nodes in dependency order. Parallelism can come later once the stage boundaries are clear.

`Context` is the in-memory working state for one pipeline run. It holds prior stage outputs so later stages can use them. If resumability is needed, the durable form of that state should be persisted outside memory, then reloaded to reconstruct context and continue from the last failed stage.

The current structure is now flatter and cleaner: pipeline-wide objects live directly under `pipeline/`, not under a nested `pipeline_objects/` folder. The main architectural pieces now are `pipeline.py`, `dag_scheduler.py`, `context.py`, `scip_indexers/`, `models/`, `worker/`, and eventually `stages/`.
