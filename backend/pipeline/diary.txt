What I learned today

The original hand-rolled language parsing / import-normalization direction was getting too deep into compiler and language-server problems. The core insight was that parsing code is only half the problem; semantic resolution is where the real pain starts. Because of that, the pipeline direction shifted toward using SCIP as the semantic substrate instead of continuing to own all low-level cross-language resolution logic manually.

SCIP is language-agnostic as a format, but language-specific in how the data is produced. That means the repo can still support multiple languages, but each language family gets its own indexer wrapper. For now, the important ones are Python and the TypeScript ecosystem. The `scip_indexers/` package exists to wrap those CLIs and normalize how the rest of the pipeline calls them.

The pipeline itself should stay job-oriented: frontend enqueues one repo-analysis job, backend worker consumes it, and one pipeline run handles the full analysis. Internally, the pipeline is made of stages/steps. Those stages should be idempotent, and the pipeline should own retry / failure handling. The queue should stay at the whole-pipeline level for now rather than making every stage its own queued task.

If the project becomes more advanced, the right execution model is a DAG of stage dependencies, not just one flat ordered list forever. The important concept is that stages declare dependency relationships, the scheduler builds the graph from those declarations, validates it, and runs ready nodes in dependency order. Parallelism can come later once the stage boundaries are clear.

`Context` is the in-memory working state for one pipeline run. It holds prior stage outputs so later stages can use them. If resumability is needed, the durable form of that state should be persisted outside memory, then reloaded to reconstruct context and continue from the last failed stage.

The current structure is now flatter and cleaner: pipeline-wide objects live directly under `pipeline/`, not under a nested `pipeline_objects/` folder. The main architectural pieces now are `pipeline.py`, `dag_scheduler.py`, `context.py`, `scip_indexers/`, `models/`, `worker/`, and eventually `stages/`.

What I learned today (Apr 28)

Spent a long stretch fighting two pipeline bugs and one deeper conceptual confusion. First, scip-python does not actually emit `SymbolInformation.kind` — verified directly against their source (`packages/pyright-scip/src/treeVisitor.ts`) and against an open PR (#164) titled "Add display_name and kind to output SymbolInformations." So every symbol comes back as `UnspecifiedKind` not because we're reading it wrong, but because the field is never written. The fallback we wrote (`_kind_from_symbol_id`) parses the SCIP descriptor suffix grammar from `scip.proto` lines 162-170, which is the canonical way to recover kind. The SCIP spec itself states "If two symbols have different Suffixes, they should have different Kinds" — so the descriptor suffix is the official 1:1 mapping. This isn't a hack; it's what Sourcegraph's own backend does today. Same story for `enclosing_range`: scip-python only emits it for function and class definitions, never for parameters, terms, modules, or annotations. That's why the AST-based end-line expansion in `BuildChunks` is load-bearing for Python — without it, def occurrences would have `start == end` for everything except funcs/classes.

The bigger bug in our code was that the descriptor parser had `)` mapping to `Method` instead of `Parameter`, which caused all 3000+ Flask parameters to be classified as methods and chunked. Fixing that (plus `:` → Meta and adding `!` → Macro) dropped the chunk count from 3642 to 1581 — the right ballpark for Flask. The flow `_CHUNK_KINDS` filters down to {Class, Function, Method, Constructor, Interface}, so the misclassification was leaking parameters into the chunked set.

Built `test_embed_chunks.py` after fixing the chunking. The point of that test isn't to validate the embedding model — `sentence-transformers` is battle-tested code, not ours. The point is to validate the thin wrapper: the `zip(chunks, embeddings)` alignment, the dim, the pre-normalization, the empty-context short-circuit. Module-scoped fixture so the model loads once. All six tests pass in ~2.4s after warm load.

Bigger shift today was on product direction. Started the day thinking about a "code chatbot," realized that's a saturated market (Cursor, Copilot, Cody, Phind) and any portfolio piece in that space looks like a clone. Walked through several pivots — curriculum generator, then quizzes only — and finally landed on what's actually defensible: an adaptive learning system for codebases. Not "Quizlet for repos" framed as quiz generation, but framed as a personalized learning model that happens to use quizzes as the surface.

The key realization, prompted by another agent's feedback and pressure from myself: "deterministic correctness from real graph data" is interesting infrastructure but it isn't the headline product feature. Cursor produces correct code too — correctness alone isn't a moat. What chat agents fundamentally cannot do is **maintain a persistent model of what the user knows**. Quizlet, Anki, and Duolingo don't win because their flashcards are technically better than ChatGPT's; they win because they apply learning science: per-user knowledge state, adaptive question selection, spaced repetition (FSRS/SM-2), misconception detection across sessions. None of those are possible in a stateless chat. That's where the moat actually lives.

The pipeline I've built feeds directly into this. Every quiz question gets tagged with the cluster it belongs to, the symbols it references, the concept it tests, and a difficulty level. After each question, the user's mastery vector updates per-tag. The adaptive selector picks the next question to maximize learning progress: prioritize weak topics, schedule spaced reviews, surface targeted follow-ups when the user gets a related concept wrong. The graph data (call graph, dependency graph, clusters) stays valuable because it makes the *tagging* meaningful — "you're weak in `app_context`" is concrete because the cluster came from real Louvain clustering of the actual file dependencies, not LLM hallucination. The "show the source data" affordance — exposing the SCIP/graph evidence backing each correct answer — becomes a trust feature, not the headline.

On storage: rejected SQL for the indexed artifacts. The data is ~13MB per repo, fits trivially in memory, the access patterns (point lookups, graph traversal, vector search) are all things SQL is mediocre at. Going with JSONL + JSON + numpy `.npy` files on disk under `data/indexed/<repo>/`, mmap'd into memory once at backend startup, with Redis pub/sub `index:ready` events to hot-load new repos. SQL still has its place — operational state (users, sessions, attempts, per-tag mastery, review schedule) lives in Postgres because that's hot, mutable, queryable. Polyglot persistence: each tool does what it's good at.

LLM choice: Gemini 2.0 Flash free tier. 15 RPM, 1500 requests/day, 1M tokens/day — enough to summarize 1500 chunks per repo, generate cluster + repo summaries, and produce a quiz bank, all for $0. Aggressive content-hash caching means re-indexing the same repo costs nearly nothing. The user is solo-bootstrapping this project, so any architecture that introduced real LLM cost was a non-starter.

Five new pipeline stages were scaffolded today as empty `BaseStage` shells, ready to fill in: `SummarizeChunks`, `SummarizeClusters`, `SummarizeRepo`, `GenerateQuizBank` (renamed from the earlier `GenerateCurriculum`), and `PersistContext`. The DAG scheduler resolves them into ten parallelism-friendly levels, with the three summarization stages able to run concurrently. The locked stage order matches the locked scope: summarize after embedding, generate quiz bank after summaries, persist last as a single atomic write so the backend never sees half-indexed state.

Lock-in for the build: stop pivoting. Quiz product, framed as adaptive learning, with a recruiter-grade live demo URL. Pre-index a handful of popular Python repos (Flask, FastAPI, Requests, Click) so the demo has instant content with no upload wait. Three demo-critical features: source-backed answers (every question can show its graph evidence), diagram-picker question type using real call graphs, and the mastery-coverage heatmap. Persistence and spaced repetition make it a real product but aren't required for the 60-second demo. Build the wrapper, not the chat.

What I learned today (May 4)

Plugged some gaps in pipeline 1's persistence before turning to pipeline 2 design. The chunk call graph (chunk.calls / chunk.called_by) was being computed in BuildChunks but PersistContext silently dropped it. The file-level dependency graph from BuildDependencyGraph was getting consumed by ClusterGraphs and then discarded. The occurrences table was throwing away role_flags (the raw SCIP bitmask) and enclosing_symbol_id (the field the call graph is literally built from). The README was being read from disk by SummarizeRepo at run time and never persisted, leaving pipeline 2 with a hidden disk dependency. All of those got fixed: chunk edges as JSON arrays on the chunks row, file dependencies as a JSON dict on the files row, the two missing occurrence columns, and a single source-of-truth README load in ParseRepo that everything downstream uses.

The interesting lesson on the schema round was about reflexive normalization. First instinct was to add three new edge tables — chunk_calls, chunk_called_by, file_dependencies — because that's what textbook SQL normalization says to do for set-valued fields. Got called on it. The data is loaded fully into memory at pipeline 2 time, the access pattern is iteration not SQL JOIN, and the FK constraints I wanted for integrity weren't even possible because callees aren't all in the symbol table. Flattened everything into JSON columns on the existing rows. The principle worth keeping: don't normalize reflexively. Ask whether the queries justify the normalization, and prefer JSON-on-row when the data is read whole and iterated in code anyway.

Hoisted the Pipeline definition out of pipeline.context into pipeline/core/pipeline.py as an ABC with __init__(scheduler, executor) defaults to DagScheduler / SequentialExecutor, and an abstract run. ContextPipeline now inherits from it; QuestionPipeline is an empty skeleton ready to fill. Tiny refactor, but it makes the second pipeline a real architectural commitment instead of a copy-pasted file waiting to drift.

Most of the afternoon was on pipeline 2's architecture, and the design walked through several wrong shapes before landing on the right one. Started from "give the LLM tools and ask for 50 questions" — that's the chat-product mental model, wrong here because we already know the queries at generation time. RAG and agentic tool-use are for runtime when the user query is unknown; at batch time we walk the data deterministically and feed targeted per-unit prompts. Conversely "stuff the whole repo into one prompt and ask for 50 questions" is also wrong — context degrades quality past ~32K tokens (lost-in-the-middle), coverage skews to whatever's near the start/end, and big repos won't fit anyway. The right primitive is per-cluster prompts with bounded context — the same shape pipeline 1's summarize stages already use, just emitting questions instead of summaries.

The biggest single reframe was on question types. Originally I was going to define MCQ, DragAndDrop, FillInBlank, etc. as generation-time outputs and have the LLM pick a type. Wrong layering: the *fact* is what's stable, the *form* is a render-time choice. Generate (prompt, correct_answer, distractors[6-8 of them], solution) per question, and let the frontend render the same row as MCQ today, fill-in-blank tomorrow, true/false the day after. Same bank multiplies into different question feels. The 6-8 distractors instead of 3 is the same insight at a smaller scale — sample 3 at serve time, exclude the one the user fell for last presentation, and a single question becomes a misconception probe. Quizlet Learn essentially does this; copying the pattern gets the variety without any extra LLM cost. Also dropped the planned ValidateQuestions and DedupeQuestions stages — same-model judges share the same blind spots as the generator, and per-concept generation doesn't dupe itself within a single call. Replaced with cheap structural validation at persist time (in-range correct_answer index, no duplicate distractors, valid source_refs).

The hierarchical clustering question wasted real time because of two wrong design suggestions before the right one. First idea: "run Louvain at multiple resolution=γ values" — γ adjusts the modularity bias to fight the resolution-limit problem (Fortunato & Barthélemy 2007), which is totally orthogonal to producing a layer hierarchy. Second idea: "build agglomerative Louvain on a meta-graph" — that's just renaming what Louvain's own phase-2 supernode reduction already does internally. The actual answer was always one function call: `community_louvain.generate_dendrogram(graph, weight="weight")`. Louvain is intrinsically multi-level — every phase-1+phase-2 iteration produces a coarser partition, and the library exposes those levels directly via partition_at_level. The lesson: when an algorithm has an inherent hierarchical structure, check whether the library already exposes it before inventing a wrapper.

For per-cluster prompt input, settled on a hybrid that always includes the cluster summary plus every chunk's summary in scope (cheap — ~30 tokens each), then adds full chunk.code for the top-K most central chunks where K is bounded by remaining token budget. "Central" = sum of len(chunk.calls) + len(chunk.called_by), the load-bearing nodes in the call graph. Gives the LLM the full structural picture (no chunk is invisible) plus concrete code anchors (so questions can ground in real implementation). For a Flask layer-1 cluster of ~5 files / 30 chunks / 10 central-chunk codes loaded, this lands around 8K tokens — well within budget at every layer of the dendrogram.

The adaptive learning surface that does the most work for the least implementation cost is Leitner-style spaced repetition keyed to clusters. Per-user × per-cluster box level (1..5) with growing review intervals — promote on right, demote to box 1 on wrong, surface overdue low-box clusters preferentially. The static bank plus per-user box state combine into something that *feels* adaptive without per-question state tracking. Pure serving-layer concern, not a pipeline concern. Rejected FSRS — too complex for v1, Leitner is canonical Anki and well-understood.

Stylistic cross-cutting got parked. The structural hierarchy from generate_dendrogram captures structural cross-cutting (auth subsystem touching multiple modules will surface as a single cluster at some upper level), but it doesn't capture stylistic patterns — "the codebase uses content-hash caching everywhere" isn't a Louvain cluster because the caching isn't structurally co-located. Considered per-chunk style tagging and rejected it as too syntactic; "uses async" is a useless tag because it lumps unrelated things together by a shared keyword rather than a shared intent. The right shape, when this comes back, is pattern mining at SummarizeRepo time with intent-named patterns ("async with rate-limiting for LLM calls", not "async") and recurrence enforcement (must appear in ≥2 clusters). Not blocking pipeline 2 v1.

Also rejected MCP. It's the right standard for the wrong problem here — MCP is for exposing tools to a third-party host (Claude Desktop, Cursor) over a transport. We're a self-contained batch pipeline calling the OpenAI API directly. Direct structured outputs lose nothing and add no server-protocol-serialization tax.
